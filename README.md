Booking System — це мікросервісна платформа для управління бронюванням ресурсів, яку можна адаптувати під бронювання готелів, кімнат, товарів, обладнання або послуг.
Система підтримує два типи користувачів:
USER — реєстрація, авторизація, перегляд доступних ресурсів, створення бронювання, перегляд статусів оплат, отримання повідомлень.
ADMIN — керування ролями, користувачами, доступ до аналітики та системних даних.
Система включає: аутентифікацію через JWT, асинхронні події через Kafka, кешування через Redis, стійкість до збоїв через Resilience4j, централізоване логування через власний Starter, OpenFeign для міжсервісної взаємодії та Spring Cloud для конфігурації та сервіс-дискавері.

Мікросервіси:
1. auth-service - відповідає за реєстрацію, логін, оновлення даних користувачів та видачу JWT-токенів. Реалізовано: хешування паролів через BCrypt, фільтри JWT, перевірка ролей. Публікує події user.registered у Kafka.
2. admin-service - доступний тільки ADMIN-користувачам.
Використовує OpenFeign для взаємодії з auth-service (отримання списку користувачів, зміна ролей, видалення).
3. booking-service - основна бізнес-логіка бронювання ресурсів, де є можливість створення, скасування, фільтрація бронювань, та інше.
Використовується Spring Data JPA Specifications — для гнучкої фільтрації за датою, статусом, itemId, userId тощо. Після створення або зміни бронювання надсилає подію booking.created у Kafka. Логування та обробка помилок реалізовані через спільний стартер.
4. inventory-service (item-service) - відповідає за ресурси, що можна бронювати: кімнати, продукти, послуги тощо. Реалізовано CRUD-операції. Підтримує Spring Data JPA Specifications для динамічної фільтрації.
Використовується Redis як кеш для швидкого доступу до часто змінюваних ресурсів.
Публікує події змін у Kafka та логування через стартер.
5. payment-service - імітація платіжної системи. Приймає запити на оплату бронювання, виконує просту бізнес-логіку перевірки та надсилає подію payment.success у Kafka.
6. notification-service - слухає Kafka-топіки user.registered, booking.created, payment.success. Формує повідомлення та надсилає користувачам.
7. api-gateway – це центральна точка входу в систему, маршрутизація запитів, фільтрація JWT. Валідує jwt-token і потім дає доступ до внутрішніх сервісів через маршрутизацію.
8. config-server - централізоване управління конфігами для всіх сервісів. Зчитує application.yml для кожного сервісу.
9. eureka-server – сервіси автоматично реєструються і знаходять один одного.
    
Додатковий функціонал
1.	Підключено Prometheus + Grafana для моніторингу (метрики, статуси сервісів).
2.	Успішні запити, обгорнуті у Resilience4j(retry, timeouts, circuit breaker для стабільної міжсервісної комунікації.).
3.	Централізований логувальний стартер. Усі мікросервіси логують однаково: за допомогою AOP, доставка логів відбуваєтться через Kafka, усі мають єдиний формат JSON-подій, і передаються в ELK, для легкого пошуку та візуалізації, щоб легше було знайти помилку і виправити. 
Такий підхід повністю відповідає принципу DRY, зменшує обсяг коду в кожному сервісі та спрощує підтримку.
4.	Усі сервіси, а також Postgres, Kafka, Redis, UI-інтерфейси для них, Prometheus, Grafana та ELK-стек були підняті через Docker Compose.
Для оптимізації збірки кожного сервісу я використав multi-stage Dockerfile, що дозволило зменшити розмір образів, пришвидшити запуск контейнерів та знизити використання пам'яті.
5.	Майже для кожного мікросервісу реалізовано інтеграційні тести з Testcontainers (PostgreSQL, Kafka, Redis) та unit-тести, що забезпечує стабільність змін і спрощує рефакторинг.
6.	Для ручного тестування API використовував Postman.
